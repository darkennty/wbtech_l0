# Демонстрационный сервис с Kafka, PostgreSQL, кешем

## Описание

Данный проект представляет из себя небольшой микросервис на Go с использованием базы данных и очереди сообщений. Сервис получает данные заказов из очереди (Kafka), сохраняет их в базу данных (PostgreSQL) и кэширует в памяти для быстрого доступа.

Что было сделано:

### База данных:

1. Cоздание новой базы данных для сервиса

2. Настройка пользователя согласно заданным переменным окружения.

3. Создание таблиц с помощью инструмента `migrate`

### Сервис:

1. С помощью инструмента **swagger** создан простейший интерфейс для отображения полученных данных по ID заказа.

2. Сервис подключается к топику и подписывается на канал сообщений.

3. Сервис сохраняет полученные данные в БД: при приходе нового сообщения о заказе, сервис парсит его и вставляет соответствующую запись в базу данных (PostgreSQL).

4. Реализовано кэширование данных в сервисе: сервис хранит последние полученные данные заказов в памяти локально, чтобы быстро выдавать их по запросу. При перезапуске кэш восстанавливается из БД.

5. Действия, происходящие в ходе работы сервиса, а также все ошибки, логируются.

6. Сервис запускает HTTP-сервер для выдачи данных по ID.

   Пример запроса: `GET http://localhost:8081/order/<order_uid>`

7. По эндпоинту `/swagger/index.html` доступен простой веб-интерфейс — страница, где можно ввести ID заказа и получить информацию о нём, обращаясь к вышеописанному HTTP API.


Итого:

1. Сервис подключается к брокеру сообщений (Kafka) и обрабатывает сообщения онлайн (можно написать скрипт-эмулятор отправки сообщений).

2. Кеш ускоряет получение данных (например, при повторных запросах по одному и тому же ID).

3. HTTP-сервер возвращает корректные данные в формате JSON.

## Запуск сервиса

Перед запуском сервиса необходимо в корневой папке проекта создать файл .env, пример для которого есть в репозитории (файл .env.example) и прописать переменные окружения для сервиса. Те же инструкции необходимо сделать с файлом config.yml, только уже в папке configs.

Для запуска всех необходимых для сервиса элементов необходимо собрать и запустить контейнеры:

```
docker-compose build
docker-compose up -d
```

Далее в папке с проектом необходимо установить требуемые зависимости с помощью команды:
```
go mod download
```

После чего нужно запустить сам сервис, выполнив команды:

```
go build ./cmd/app/main.go
go run ./cmd/app/main.go
```

## Структура проекта

### `cmd/app`

Содержит файл main.go, который является точкой старта работы сервиса. В нём настраиваются переменные окружения, необходимые для работы сервиса, а также данные из файлов конфигураций. Программа вызывает метод `Run()` пакета `internal/app`.

### `configs`

Содержит файлы конфигурации для правильной работы сервиса.

### `docs`

Содержит необходимые файлы для работы инструмента **swagger**.

### `internal`

Вся бизнес-логика сервиса находится в папке `internal`.

#### `internal/api`

- `internal/api/handler` - создание обработчика http-запросов, реализация эндпоинтов. Использутся фреймворк **gin**. Сопровождается тест-кейсами.
- `internal/api/server` - создание сущности сервера и реализация функции запуска и остановки http-сервера.

#### `internal/app`

Содержит функцию, вызываемую из `cmd/app/main.go`. Осуществляется запуск сервиса:
- Подключение к базе данных
- Создание репозитория (реализует взаимодействие с **PostgreSQL**)
- Создание и прогрев кэша
- Создание сервиса (работает с репозиторием, не зависит от реализации работы с БД)
- Создание обработчика http-запросов
- Запуск http-сервера
- Настройка и запуск kafka-консьюмеров
- Реализация *Graceful shutdown*

#### `internal/caches`

Реализует кэширование: содержит описание структуры кэша, методы работы с кэшем (Get(), Set(), WarmCache()). Сопровождается тест-кейсами.

#### `internal/kafka`

Производит работу с брокером сообщений kafka. Используется библиотека `github.com/confluentinc/confluent-kafka-go/v2`.

- `internal/kafka/consumer` - описание консьюмера, реализация методов его запуска и остановки.
- `internal/kafka/handler` - создание обработчика сообщений из брокера. Сопровождается тест-кейсами.

#### `internal/model`

Описывает модель принимаемых из брокера сообщений данных.

#### `internal/repository`

Описывает абстракцию "Репозиторий", отвечающую за взаимодействие с базой данных (в данном проекте - с PostgreSQL).
Содержит методы подключения к БД, . Сопровождается тест-кейсами.

#### `internal/service`

Описывает абстракцию "Сервис", которая взаимодействует с репозиторием и отдаёт данные обработчику http-запросов. С помощью данной абстракции достигается независимость обработчика от конкретной технологии реализации базы данных (обработчик http-запросов обращается к сервису и получает нужные данные, без знания того, каким именно образом и с помощью какой технологии эти данные были получены).

#### `internal/testdata`

Содержит тестовые данные для работы тест-кейсов проекта.

### `migrations`

Содержит файлы для миграции БД.

### `.env`

Файл с переменными окружения для работы проекта (имя и пароль пользователя PostgreSQL, порты и пр.).

### `.docker-compose.yml`

Описывает сборку и развёртывание контейнеров с образами Kafka, PostgreSQL и migrations (для миграции при создании БД).
